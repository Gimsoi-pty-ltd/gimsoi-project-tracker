# Gimsoi Agent Guardrails

These rules are hard constraints. They are not suggestions. No workflow step,
test result, or user phrasing overrides them.

---

## Actions Requiring Explicit User Confirmation

The agent must STOP and display the exact command it intends to run, then wait
for the user to type "Confirmed" before executing any of the following:

| Action | Why |
|---|---|
| `prisma migrate deploy` | Irreversible on production DB |
| `npx prisma db push --force-reset` | Destroys all data |
| `npm install <new-package>` | Adds undeclared dependency |
| `git add` + `git commit` | Local history change |
| Any modification to `prisma/schema.prisma` | Schema drift risk |
| Any modification to `server.js` | Affects all routes and middleware |
| Any modification to `middleware/` | Global request pipeline |

---

## Git Push Is Permanently Prohibited

The agent **must never** run any of the following commands or any variation of them:

```
git push
git push origin
git push --force
git push --force-with-lease
git push origin <branch>
git push upstream <branch>
```

This rule has **no exceptions**:
- Not after a passing `npm run test:api` run
- Not at the end of the feature cycle
- Not during the `/pre-deploy` workflow
- Not if the user says "ship it", "deploy it", "push it", "send it", "go live", or any similar phrasing

If the user says anything that could be interpreted as a request to push to a remote,
the agent must respond with:

```
ã GIT PUSH INTERCEPTED
I do not run remote Git operations. If you want to push, run this yourself:

  git push origin <branch-name>

Type "Confirmed" only if you want me to assist with something else related to git (e.g. staging or committing locally).
```

Local `git add` and `git commit` are allowed **only when explicitly instructed** by the user.

## Prisma-Specific Footguns

### Always run `prisma generate` after a schema change
Forgetting this causes the Prisma client to be out of sync with the schema.
The app will crash at runtime with a `PrismaClientInitializationError`.
The phase that modifies `schema.prisma` must include `npx prisma generate` as
its final step before `npm run test:api`.

### P2025 ã Record not found
`PrismaClientKnownRequestError` with code `P2025` is thrown when a `findUnique`,
`update`, or `delete` targets a record that does not exist. This must be caught
explicitly in service files and converted to a 404 application error.
Letting it bubble to Express as an unhandled error produces a 500.

### P2002 ã Unique constraint violation
Code `P2002` is thrown on duplicate `email`, `slug`, or any field marked `@unique`
in the schema. Must be caught and converted to a 409 application error.
Check `error.meta.target` to identify which field triggered the constraint.

### Never use `prisma.$queryRaw` or `prisma.$executeRaw`
Raw SQL bypasses Prisma's type safety and multi-tenant isolation logic in
`task.service.js`. If a query cannot be expressed in Prisma's query API,
stop and surface the constraint to the user ã do not write raw SQL.

### Migration naming convention
Migration names must be snake_case and describe the change:
`add_task_priority_field`, `rename_profile_pic_to_avatar_url`, `add_project_end_date`.
Never use `migration1` or timestamp-only names.

## AppSail Deployment Footguns

### nodemon in production
`package.json` has `"server": "nodemon server.js"` as a dev convenience.
AppSail must use `"start": "node server.js"`. Confirm the Catalyst dashboard
start command before every deploy. If `nodemon` is the configured start command,
deployment is blocked.

### Missing env vars
Catalyst AppSail does not read `.env` files. All variables from `.env.example`
must be set in the Catalyst environment config. Missing vars cause silent startup
failures that are only visible in AppSail logs, not in `catalyst deploy` output.
Required vars (from `.env.example`): `DATABASE_URL`, `JWT_SECRET`, `CSRF_SECRET`,
`CORS_ORIGIN`, `NODE_ENV`, `PORT`.

### ESM/CJS conflict on AppSail
The package has `"type": "module"`. Any file with `.cjs` extension or a
`require()` call will cause a `ERR_REQUIRE_ESM` crash at startup.
This error appears in AppSail logs as a silent exit ã the health check will
time out with no obvious error message in the `catalyst deploy` output.

## Scope Discipline ã Files Off-Limits Without Explicit Instruction

The following files must not be modified unless the current phase plan explicitly
names them:

- `server.js`
- `middleware/` (all files)
- `prisma/schema.prisma`
- `lib/generated/` (all files ã auto-generated by Prisma)
- `package.json` (unless adding a flagged-and-approved new dependency)
- `package-lock.json` (only changes when `npm install` runs)
- `.env` / `.env.example`
- `playwright.config.js`
- `tests/setup/` (unless QA persona is active and checklist is approved)

## Escalation Threshold (Binary, Not Subjective)

When a phase fails `npm run test:api`, the fix is classified as follows:

| Condition | Classification | Action |
|---|---|---|
| Fix requires changes to **1 file**, **same layer** | **SIMPLE** | Fix inline, re-run `npm run test:api` immediately |
| Fix requires **>1 file** OR touches **>1 layer** | **COMPLEX** | STOP ã trigger `/escalate`, generate sub-plan, wait for "Approved" |

There is no middle ground. Cross-layer means: controller ã service, service ã route,
route ã middleware, any combination of the above.
